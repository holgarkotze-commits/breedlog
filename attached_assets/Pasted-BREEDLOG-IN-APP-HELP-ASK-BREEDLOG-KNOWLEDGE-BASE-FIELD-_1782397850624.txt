BREEDLOG — IN-APP HELP, ASK BREEDLOG KNOWLEDGE BASE, FIELD TEST ISSUE REPORTING, PWA INSTALL-FIRST FLOW, AND SAVED ACCESS CODE SYSTEM

Objective:
Before generating the final full documentation pack, first upgrade BreedLog so the app itself contains the documentation/help knowledge, and the Ask BreedLog agent can answer questions from that knowledge and from the relevant app data available to the authenticated user.

This must be production-mode work. We are entering final real-world testing with 20+ users. Do not make demo-only shortcuts. Do not break completed export work, active herd count work, Records tab work, sync logic, access code logic, or existing animal data workflows.

Current accepted baseline:

* Latest accepted BreedLog HEAD: 7d62e1e on main, working tree previously clean.
* Export standardization accepted.
* Active animal count accepted.
* Light/dark contrast accepted.
* JSON removed from normal exports accepted.
* My Herd opens minimized accepted.
* Records tab Round 1 implemented/build-clean but may still need final production proof if not already committed.
* STITCH WORX signature must be used where appropriate.

This new task has six main deliverables:

1. In-app Help / Info documentation hub.
2. Ask BreedLog knowledge base integration.
3. Question-chip hardening so suggested questions always have answers.
4. Field-test issue reporting with email notification and admin-panel tracking.
5. PWA install-first onboarding for phone and desktop users.
6. Safe “remember my access code” / saved-login convenience flow.

Important privacy/data rule:
Ask BreedLog may answer from:

* Public app help/documentation content.
* The authenticated user’s own workspace/farm data.
* Records, exports, animals, health, breeding, productivity, and settings data that the current user is allowed to see.

Ask BreedLog must not:

* Leak data between different access codes/workspaces.
* Expose secrets, API keys, admin-only internals, server logs, or database credentials.
* Invent unsupported answers.
* Answer from another user’s farm data.
* Pretend a feature exists if it does not exist.

PHASE 8 — PWA INSTALL-FIRST ONBOARDING

Important user requirement:
Users should not simply open the browser link on phone/desktop and immediately enter their access code in Chrome/Opera/Edge/Safari browser mode. They must be guided to install BreedLog as an app first, then use their access code from the installed app/PWA.

Browser limitations:
Different browsers expose the install button differently. Chrome/Edge desktop often show an install icon in the address bar. Android Chrome may show install prompts or menu options. iOS Safari uses Share → Add to Home Screen. The app cannot perfectly control every browser address bar icon, but it can detect browser mode, show a strong install-first screen, and guide the user with animated arrows/instructions appropriate to the platform.

Required implementation:

1. Detect whether app is running as installed PWA/standalone:

   * display-mode: standalone
   * navigator.standalone for iOS where applicable
   * other safe PWA detection already used in the repo if present

2. If user opens BreedLog in normal browser mode:

   * Show an install-first onboarding screen before access-code entry.
   * Do not show the normal access-code entry as the first primary action.
   * Strongly guide user to install the app first.
   * Use animated arrows or attention indicators pointing toward the install area/instruction.
   * Use platform-specific instructions.

3. Desktop Chrome/Edge:

   * Show instruction: click the install icon in the browser address bar.
   * Show animated arrow/indicator toward top/right area.
   * If beforeinstallprompt is available, show an “Install BreedLog App” button that triggers the native install prompt.

4. Android Chrome/Edge:

   * If beforeinstallprompt is available, show “Install BreedLog App”.
   * If browser menu is needed, guide user to menu → Install app / Add to Home screen.
   * Use animated arrow or highlighted instruction.

5. iPhone/iPad Safari:

   * Show instruction: tap Share, then Add to Home Screen.
   * Provide clear step-by-step instructions.
   * Do not falsely show a Chrome-style address-bar install icon on iOS Safari.

6. Unsupported browsers:

   * Show instructions to use Chrome/Edge/Safari appropriate to device.
   * Allow a clearly controlled fallback only if necessary for testing/admin, but normal users should be pushed to install.

7. After successful install / standalone launch:

   * Show normal access-code screen.
   * User can enter access code.
   * User can optionally save the code for this device.

Required install-first screen content:

* BreedLog logo/name.
* “Install BreedLog before signing in.”
* Short reason: better offline use, faster access, app-like experience, fewer login issues.
* Platform-specific installation steps.
* Animated pointer/arrow or visual guidance.
* Button: “I have installed BreedLog — continue”.
* Optional fallback: “Continue in browser for testing/admin” only if explicitly enabled or behind admin/test flag.

Important:
Do not permanently lock all users out if PWA install detection fails. Provide a safe admin/test bypass or a controlled fallback, but default normal user flow must strongly enforce install-first.

PHASE 9 — SAVED ACCESS CODE / REMEMBER LOGIN FLOW

User requirement:
Users forget their access codes. The app must provide a reliable way to save the access code on the device so they do not need to retype it every time.

Required behavior:

1. Add “Remember this access code on this device” option on the access-code screen.
2. If checked, save the access code locally on that device after successful login.
3. On next launch, prefill the access code field with the saved code.
4. User can tap “Continue” without retyping.
5. User can clear/remove saved access code.
6. Logging out should not automatically delete saved code unless the user chooses “Forget this device/code”.
7. If user chooses not to save, do not store the code.
8. If saved code becomes invalid, show a clear message and let the user re-enter a new code.
9. This must work on both installed PWA and desktop installed PWA where storage is available.

Security rules:

1. Do not save admin PINs unless there is already a separate secure admin-auth design.
2. Do not send saved codes to analytics.
3. Do not expose saved codes in logs.
4. Store only what is needed.
5. Prefer local device storage. Do not create a server-side plaintext password vault.
6. If the app already has a secure credential helper, use it.
7. Provide “Forget saved code” control in Settings or access screen.

Suggested local storage key names should be clear and namespaced, for example:

* breedlog.rememberAccessCode
* breedlog.savedAccessCode

Do not store unrelated user data in this flow.
