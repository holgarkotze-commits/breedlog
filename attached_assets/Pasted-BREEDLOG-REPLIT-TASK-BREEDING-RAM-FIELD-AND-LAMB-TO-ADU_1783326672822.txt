GOAL 1 — ADD SEPARATE “BREEDING RAM” STATUS FIELD

Problem:
BreedLog currently allows a ram to have a class/type/classification, but Haka needs a separate field to mark whether a ram is a breeding ram.

Important logic:
“Breeding Ram” is NOT the same as class/type.
A ram can be:

* Stud ram + breeding ram
* Commercial ram + breeding ram
* Stud ram + not currently breeding
* Commercial ram + marketable ram
* Marketable ram
* Cull/not retained

Required:

1. Add a separate ram status/usage field for rams.
2. Do not replace the existing class/type/classification field.
3. The new field must be available:

   * when adding a new ram
   * when editing an existing ram
   * when a ram lamb becomes old enough and moves/admitted into the Rams section
4. Suggested field name:
   ramBreedingStatus
5. Suggested values:

   * breeding_ram
   * marketable_ram
   * not_selected
   * unknown
6. UI labels:

   * Breeding Ram
   * Marketable Ram
   * Not Selected
   * Unknown

Acceptance:

* A ram can be classed as “Stud” and also marked as “Breeding Ram.”
* A ram can be classed as “Commercial” and also marked as “Breeding Ram.”
* Existing class/type data must still display and export correctly.
* New breeding status must persist after save/reload.
* New breeding status must be available in add/edit ram forms.
* Existing animals with no value must safely show Unknown or Not Selected.

GOAL 3 — IMPLEMENT AUTOMATIC LAMB-TO-ADULT SECTION LOGIC IF NOT ALREADY 100% CORRECT

Haka’s required rule:
When a lamb reaches the BreedLog age/rule for leaving the Lambs section, it must automatically move into the correct adult/current section and all section counts must sync.

Required behaviour:

1. A lamb that has come of age must no longer appear under the Lambs tab.
2. The Lambs count must decrease accordingly.
3. If the animal is female and retained/active, it must appear under Ewes.
4. If the animal is male and retained/active, it must appear under Rams.
5. If the animal is culled/sold/deceased/inactive, it must appear only in the correct inactive/culled/sold section, not as an active lamb.
6. The moved animal must keep its complete data:

   * animal ID/tag
   * birth date
   * sex
   * sire/dam
   * birth weight
   * weaning data
   * health records
   * breeding/genetics/bloodline data
   * images
   * notes
   * all linked records
7. The move must be classification/status logic only, not destructive migration.
8. Do not duplicate animals.
9. Do not create a new animal record when ageing occurs.
10. Do not lose lamb history.

Preferred implementation:
Use derived section membership from age + sex + status + retained/cull/sold state, unless the existing architecture already has a safe canonical section field.
If there is already a canonical section field, update it safely and test all counts.

GOAL 4 — RAM LAMBS BECOMING RAMS

When a ram lamb becomes old enough and is admitted/moved into the Rams section:

1. It must no longer show in Lambs.
2. It must show in Rams.
3. Rams count must increase by 1.
4. Lambs count must decrease by 1.
5. It must retain all lamb history and performance data.
6. It must allow ramBreedingStatus selection:

   * Breeding Ram
   * Marketable Ram
   * Not Selected
   * Unknown
7. It must still keep its separate class/type/classification field.

GOAL 5 — EWE LAMBS BECOMING EWES

When a ewe lamb becomes old enough and is admitted/moved into the Ewes section:

1. It must no longer show in Lambs.
2. It must show in Ewes.
3. Ewes count must increase by 1.
4. Lambs count must decrease by 1.
5. It must retain all lamb history and performance data.
6. It must not receive ramBreedingStatus unless shared schema requires nullable compatibility.
