BREEDLOG — INDIVIDUAL ANIMAL EXPORT PERFORMANCE DATASHEET UPGRADE

Objective:
Upgrade the individual animal PDF export so each exported animal document becomes a professional A4 portrait performance datasheet with real breeder value.

The current individual animal export does not show enough meaningful performance data. It is too basic and does not properly explain the animal’s production value, breeding value, progeny performance, growth record, health/breeding status, or overall performance rating.

This task applies to individual animal exports only:

* Individual Ram export
* Individual Ewe export
* Individual Lamb export
* Young stud ram export
* Young stud ewe export
* Meat/production animal export where applicable

Do not redesign group exports.
Do not break the completed canonical PDF export template system.
Do not change stored animal records unless a real bug is found.
Do not invent data.
Do not create fake performance values.
Do not make unrelated app changes.

Current accepted baseline:

* BreedLog export-template work is accepted and must not be broken.
* Individual exports are portrait A4.
* Canonical header/footer styling exists.
* Footer must remain fixed at the bottom of each page.
* STITCH WORX signoff remains appropriate in product/documentation areas only, not as a replacement for BreedLog branding.

PHASE 1 — AUDIT CURRENT INDIVIDUAL EXPORT DATA SOURCES

Before changing layout, audit the real data available in the app.

Inspect:

1. Individual animal export function/component:

   * `AnimalDetail.tsx`
   * Any helper used by individual PDF export
   * Family tree/pedigree export logic
   * Animal photo/image handling
   * Current section layout and page sizing

2. Animal data:

   * Animal ID/tag ID
   * Name
   * Sex
   * Breed/type
   * DOB/age
   * Status
   * Sire/dam
   * External sire/dam info if present
   * Current group/category
   * Registration/stamboek/classification fields if present
   * Notes

3. Performance data:

   * Birth weight
   * Weaning weight
   * 100-day weight
   * 270-day weight
   * Latest/current weight
   * Weight history
   * Growth intervals if derivable
   * Average daily gain if derivable
   * Any condition/performance/class fields already stored

4. Breeding/progeny data:

   * Ram progeny/lambs bred
   * Ewe lambing history
   * Lambing count
   * Lamb survival status if derivable
   * Lamb birth weights
   * Lamb weaning/100-day/270-day weights
   * Progeny sex split if derivable
   * Progeny growth averages if derivable
   * Mating group participation
   * Breeding events

5. Health data:

   * Treatments
   * Vaccinations
   * Health events
   * Disease/treatment history if present
   * Recent health status if derivable

6. Meat/production data:

   * Growth performance
   * Weight gain
   * Market readiness where derivable
   * Production status/category
   * Meat animal notes/classification if present

Only use data that exists or can be correctly derived from existing app records. Where data is missing, show “Not recorded” or “Insufficient data”, not fake values.

PHASE 2 — NEW INDIVIDUAL EXPORT PURPOSE

The individual export must answer:

1. What animal is this?
2. What is its identity and pedigree?
3. What is its growth/performance record?
4. What is its breeding value or production value?
5. For rams: how have its lambs/progeny performed?
6. For ewes: how productive is the ewe and what is her lambing record?
7. For young stud animals: what is the development/potential based on available weights, age, class, and parentage?
8. For meat/production animals: what is the production/growth value?
9. What is the final short summary of the animal?
10. What is the animal’s performance rating based on actual available data?

This must be a professional breeder datasheet, not just a basic animal profile.

PHASE 4 — ANIMAL-TYPE-SPECIFIC PERFORMANCE DATA

Do not use one weak generic section for all animals. The export must change intelligently by animal type.

A. RAM EXPORT REQUIREMENTS

For a mature ram, include:

1. Ram identity and pedigree.

2. Ram growth/performance:

   * Birth weight
   * Weaning/100-day/270-day/latest weight where available
   * Growth trend if derivable
   * Average daily gain if derivable
   * Weight performance rating if derivable

3. Breeding/progeny performance:

   * Number of lambs/progeny linked to the ram
   * Number of lambing events linked to the ram if available
   * Progeny birth weight average
   * Progeny weaning/100-day/270-day weight average
   * Progeny survival/live status if derivable
   * Progeny sex split if derivable
   * Best-performing progeny if derivable
   * Latest progeny performance period if derivable

4. Production/breeding value:

   * Show a ram performance rating based on available metrics.
   * Rating must be deterministic and explainable.
   * Do not let AI invent the rating.
   * If insufficient progeny data exists, rating must say “Insufficient progeny data”.

5. Ram summary under 500 characters:
   Example style, but use real data:
   “This ram has recorded progeny with strong average growth through weaning and 100-day stages. Current records indicate useful breeding value, with progeny performance supporting continued use. More complete weight records will improve accuracy.”

B. EWE EXPORT REQUIREMENTS

For a ewe, include:

1. Ewe identity and pedigree.

2. Ewe growth/performance:

   * Birth weight
   * Weaning/100-day/270-day/latest weight where available
   * Growth trend if derivable

3. Maternal/productivity performance:

   * Number of lambing events
   * Total lambs born
   * Singles/twins/multiple births if derivable
   * Average lamb birth weight
   * Average lamb weaning/100-day/270-day weights
   * Lamb survival/live status if derivable
   * Lambing interval if derivable
   * Most recent lambing date
   * Progeny performance summary

4. Ewe productivity rating:

   * Deterministic and explainable.
   * Based on lambing history, lamb survival, lamb growth data, and available records.
   * If insufficient data exists, show “Insufficient productivity data”.

5. Ewe summary under 500 characters:
   Must explain actual maternal value based on records.

C. YOUNG STUD RAM / YOUNG STUD EWE REQUIREMENTS

For young stud animals being raised, include:

1. Identity and pedigree.
2. Parentage quality based on available sire/dam performance.
3. Birth-to-current growth performance.
4. Weaning/100-day/270-day records where available.
5. Classification/stamboek/class fields where available.
6. Development status.
7. Potential rating based on recorded growth and parentage.
8. Clear note if data is still early/limited.

Young animal summary under 500 characters:
Must describe potential, not pretend proven breeding performance.

D. LAMB / MEAT PRODUCTION ANIMAL REQUIREMENTS

For lambs or meat-production animals, include:

1. Identity and dam/sire.
2. Birth weight.
3. Growth weights.
4. Average daily gain if derivable.
5. Weaning/100-day/270-day performance where available.
6. Health/treatment notes if relevant.
7. Meat/production readiness where derivable.
8. Production rating based on growth data.
9. Summary under 500 characters.

Do not create market-ready claims unless the data supports it.

PHASE 5 — PERFORMANCE METRICS ENGINE

Create a reusable performance metrics helper.

Recommended file:
`client/src/lib/animal-performance.ts`

Or use a repo-appropriate path.

The helper should expose functions such as:

* `buildAnimalPerformanceProfile(animal, context)`
* `getRamProgenyMetrics(animal, allAnimals, breedingRecords, weightRecords)`
* `getEweProductivityMetrics(animal, allAnimals, breedingRecords, weightRecords)`
* `getYoungAnimalDevelopmentMetrics(animal, allAnimals, weightRecords)`
* `getMeatProductionMetrics(animal, weightRecords)`
* `buildAnimalPerformanceSummary(profile)`
* `getAnimalPerformanceRating(profile)`

Rules:

1. Keep calculations deterministic.
2. Use actual data only.
3. Document formulas in comments.
4. Return “Insufficient data” where metrics cannot be calculated.
5. Do not duplicate logic inside the export component.
6. Make it testable.
7. Keep AI wording grounded in the metrics result.

Rating output:
Use simple professional rating levels, for example:

* Excellent
* Strong
* Good
* Developing
* Monitor
* Insufficient data

Do not use aggressive sales language.
Do not overrate animals with weak records.
Use transparent reasons for each rating.

PHASE 6 — SUMMARY GENERATION RULE

Every individual export must contain a summary under 500 characters.

Summary rules:

1. Must be generated from the performance profile.
2. Must be under 500 characters.
3. Must be animal-type-specific.
4. Must mention real strengths/limitations.
5. Must not mention fake data.
6. Must not claim progeny performance where no progeny exists.
7. Must not claim ewe productivity where no lambing data exists.
8. Must not claim market readiness without growth/weight support.
9. Must use clear breeder/farm language.
10. Must fall back safely when data is limited.

Implementation:
Prefer deterministic summary generation first. If using Ask BreedLog/AI service later, the AI must only polish text from a locked metrics profile and must not invent values. For this task, deterministic summaries are acceptable and safer for production.

PHASE 8 — IMAGE HANDLING AND PDF SIZE OPTIMIZATION

The individual export currently risks large PDF size if animal photos are embedded directly.

Implement image optimization before export.

Requirements:

1. Use a compressed/resized version of the animal image in the PDF.
2. Limit displayed image dimensions.
3. Limit embedded image dimensions, for example max 600px on longest side unless existing project standards differ.
4. Convert/encode as JPEG/WebP-compatible data where safe for PDF printing.
5. Use sensible quality, for example 0.70–0.80.
6. Strip unnecessary image metadata where possible.
7. Do not embed original full-resolution photos in the PDF.
8. If the image cannot be compressed, fall back to existing safe image handling and document the limitation.
9. PDF must be significantly smaller than embedding full original images.
10. The image must remain clear enough for animal identification.

Add helper such as:

* `optimizeImageForPdf(imageUrlOrData, options)`
* `getPdfSafeAnimalImage(animal)`

Testing:
Compare approximate output/image data size before and after where possible. At minimum prove that the export uses the optimized image source, not the original full-size image.
