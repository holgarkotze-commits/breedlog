Urgent BreedLog activation bug update.

This is NOT a single-code issue. Fix the universal invite-code activation/re-authorization system.

New observed behaviour:
I uninstalled the BreedLog app from my phone, opened the site again in Chrome, re-downloaded/reinstalled the app, opened it fresh, and entered the invite code with the existing testing workspace animals.

Before I even pasted the code, the access screen already showed:
“Access revoked or expired.”

After I pasted/entered the code, activation was rejected on the phone.

But when I refreshed the admin panel, the same code showed the Mobile slot as taken.

Important evidence:
The mobile slot now shows:
“Taken — activated Apr 29, 08:32”

Today is May 7. Apr 29 is the original/old activation date from before the code was revoked and later re-authorized. That means re-authorize/recover is not clearing the old mobile activation state properly, or the activation flow is reusing stale slot/device data.

Root problem to investigate:
The app is either:

1. reading stale revoked/expired state from localStorage/IndexedDB/service worker cache/client state;
2. not clearing old mobile/desktop activation bindings during re-authorize;
3. partially claiming the mobile slot even though the frontend still fails activation;
4. using old activatedAt/device-binding data from Apr 29 instead of creating a clean new activation state;
5. admin diagnostic and real activation still do not share one source of truth.

Required fix:

1. Re-authorize/recover must fully reset reusable device-slot state for every invite code, not only this code:

   * clear mobile device binding if mobile slot is released
   * clear desktop device binding if desktop slot is released
   * clear old activatedAt timestamp for released slots
   * clear stale device fingerprint/session/token binding
   * clear revoked/blocked/expired access flags
   * keep the existing workspace and animals untouched

2. Activation must be atomic:

   * do not mark mobile/desktop slot as taken unless activation fully succeeds
   * if the frontend receives failure, the backend must not leave the slot claimed
   * if the backend claims the slot successfully, the frontend must receive the access/session/workspace payload and enter the app

3. Fresh app install / fresh browser load must not show old “Access revoked or expired” before the user enters a code.
   Clear stale access error state on:

   * access page load
   * code input change
   * logout
   * reinstall/new session
   * failed previous activation reset

4. Service worker/cache hardening:
   Do not cache activation, invite-code, license, access, admin diagnostic, or workspace-auth endpoints.
   These must be network-fresh/no-store so the phone cannot reuse old revoked/expired responses.

5. Admin diagnostic must show the real current slot state:
   If a slot was re-authorized/released, it must not keep showing old Apr 29 activation data as if it is a new activation.
   New successful activation must set activatedAt to the current timestamp.

6. Add universal tests:

   * revoked code cannot activate
   * re-authorized code clears stale revoked state
   * re-authorized code clears old mobile/desktop slot binding when released
   * re-authorized code with existing workspace + animals activates successfully
   * mobile activation after re-authorize sets a new current activatedAt timestamp
   * activation failure cannot leave mobile/desktop slot marked as taken
   * frontend access page does not show stale revoked/expired message before code entry
   * activation/admin diagnostic use same shared source of truth
   * activation endpoints are not service-worker/API cached

7. Report back with:

   * exact root cause
   * exact stale fields found
   * files changed
   * tests added/updated
   * npm run check result
   * npm run build result
   * activation test result

Acceptance gate:
This is only fixed when any re-authorized active, unexpired code with a free mobile/desktop slot can activate cleanly on a freshly installed phone/PC, keeps the same workspace animals, shows a new correct activation timestamp, and never leaves a slot taken after a failed activation.
