GOAL

Add a secure, read-only AI assistant foundation to BreedLog.

The assistant must be able to answer questions using the current user’s own BreedLog workspace data only.

The assistant must be grounded in real app data, especially the Data tab/data-engine outputs, not random model guesses.

The assistant must be useful to farmers without overwhelming them.

AI PRODUCT RULES

The assistant is BreedLog Assistant.

It is not a general chatbot.
It is a read-only farm data assistant.

It may:
- explain the user’s herd data
- summarize Data tab insights
- compare rams/sires
- compare ewes
- explain lamb growth
- identify missing records
- explain data quality
- suggest what the farmer should record next
- explain breeding/lambing results
- summarize health records already captured
- answer questions about the current workspace’s animals and records

It may not:
- invent animal records
- invent weights
- invent market prices
- invent auction data
- invent health treatments
- diagnose disease as fact
- prescribe medicine
- promise profit
- give guaranteed genetic claims
- expose another user’s data
- access other workspaces
- change records
- delete records
- update records
- revoke/reactivate codes
- perform admin actions
- bypass access control

If data is missing, it must say:
“Not enough recorded data.”

If a user asks for something outside BreedLog’s data, it must explain the limitation clearly.

If a user asks for health/veterinary advice, it must respond cautiously:
- summarize recorded symptoms/events only
- recommend consulting a local veterinarian
- do not diagnose as fact
- do not prescribe dosage or medicine unless already recorded in the user’s own data as a record

If a user asks for market/auction predictions, it must respond:
- BreedLog does not yet have verified market data connected
- it can discuss readiness based on recorded weights, growth, age, and data quality
- it must not invent prices

ARCHITECTURE TO BUILD

Build this server-side flow:

Frontend AI Panel
→ POST /api/ai/chat
→ authenticated user/session check
→ workspace isolation check
→ BreedLog AI context builder
→ Gemini provider adapter
→ structured answer
→ frontend response panel

Also add:

GET /api/ai/health
GET /api/ai/suggested-prompts
GET /api/ai/context-summary

Endpoint behavior:

/api/ai/health:
- verifies server config
- returns whether AI is configured
- must not call Gemini unless a lightweight test flag is passed or unless you decide health should call provider safely
- must never reveal API key

/api/ai/suggested-prompts:
- returns grouped prompt categories and prompt templates
- must be safe static prompt metadata
- no fake answers

/api/ai/context-summary:
- returns a safe summary of what context categories are available for the current user
- must not dump full raw database
- must respect current authenticated workspace only

/api/ai/chat:
- accepts a user question
- accepts optional category
- accepts optional context target such as data-section, animal-id, breeding-group-id
- builds current workspace context
- calls Gemini server-side
- returns an answer with source notes
- never mutates app data

AUTHENTICATION AND DATA ISOLATION

Every AI endpoint must require the same auth/session/access rules as the rest of BreedLog.

The AI must only see the effective workspace/sharedUserId/account for the authenticated user.

Use the same effective user/workspace identity logic used by animals, Data tab, and sync.

Do not use access-code strings as user IDs.
Do not query global data.
Do not query all users.
Do not expose other workspace data.

If the user is not authenticated:
- return 401
- no AI call

If workspace cannot be resolved:
- return safe error
- no AI call

AI SYSTEM PROMPT

Create a strict server-side system prompt.

It must include:

You are BreedLog Assistant, a read-only livestock records assistant.
You answer only from the provided BreedLog context.
You must not invent records, animals, weights, prices, diagnoses, or treatments.
If the context does not contain enough data, say so.
You must separate facts from recommendations.
Use practical farmer language.
Keep answers concise by default.
Show which data was used.
Never expose data from other users or workspaces.
Never claim veterinary diagnosis.
Never provide medication dosage unless it is directly recorded in the user’s own records.
Never make financial or auction-price predictions without verified market data in context.
Never perform app actions or database mutations.
When giving recommendations, base them only on recorded BreedLog data and label confidence.

ANSWER FORMAT

Return structured JSON from the server to the frontend:

confidence: high | medium | low | insufficient
usedData:
warnings:
suggestedNextQuestions:

The frontend may render it as normal chat bubbles/cards.

If the model returns invalid structure, normalize safely.
