Problem:
After the UX/scroll update, the Health Plan scrolling improved, but many other BreedLog pages/windows no longer scroll with the mouse wheel over the visible content. Users are forced to drag/use the far-right page scrollbar. This is a regression. Before the Health Plan scroll fix, those other pages scrolled normally.

Goal:
Restore normal mouse-wheel and touch scrolling across the entire app while preserving the Health Plan fix.

Create hotfix branch from latest main:
hotfix-restore-global-mouse-scroll

Required diagnosis:
Inspect the changes from PR #11, especially:

* client/src/index.css
* client/src/pages/Health.tsx
* app shell / layout wrappers
* page container classes
* any scroll-pane / flex-child-scroll usage
* any overflow-hidden / h-screen / 100dvh / overscroll-behavior / min-h-0 changes

Likely issue:
A global or high-level layout/container change is trapping scroll or forcing only the far-right document scrollbar to work. Fix the root cause. Do not patch only one page.

Required scrolling behaviour:

1. Every normal BreedLog page must scroll with the mouse wheel when the pointer is over the page content.
2. Touch scrolling must work naturally on mobile/tablet.
3. Health Plan must keep its fixed top controls and improved internal scrolling.
4. Other pages must not be forced into broken internal scroll containers.
5. Long data pages must scroll normally:

   * My Herd
   * Animal list pages
   * Breeding pages
   * Genetics pages
   * Records pages
   * Data pages
   * Admin page
   * Dashboard/Home
   * Health Records
   * any page with long tables/cards/lists
6. Left sidebar/navigation must remain usable.
7. No horizontal overflow.
8. No hidden unreachable content.
9. Do not require dragging the far-right scrollbar to browse page content.

Implementation requirements:

* Restore page-level scrolling as the default behaviour.
* Only use independent scroll panes where they are deliberately needed, such as Health Plan desktop two-pane layout.
* Do not apply .scroll-pane or overscroll containment to entire page shells unless it is proven correct.
* Remove or narrow any global overflow rule that blocks normal wheel scrolling.
* Ensure main content wrappers use correct flex behaviour:

  * min-height/min-h-0 only where needed
  * overflow-y-auto only on deliberate panels
  * no accidental overflow-hidden on page roots
* If a fixed/sticky header is used, the content beneath it must still scroll with mouse wheel over the content.
* Verify dark/light mode unaffected.

Health Plan must remain correct:

* Mobile: topic accordion remains; selected topic content opens directly below the selected button and collapses when tapped again.
