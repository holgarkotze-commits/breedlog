REPLIT PROMPT — PRODUCTION HOTFIX: ADD ROBUST MANUAL SYNC & REFRESH CONTROLS

⚠️ CRITICAL CONTEXT: PRODUCTION ENVIRONMENT The BreedLog app is LIVE. These changes impact real users on mobile and web.

Constraints: Changes must work seamlessly across iOS Mobile Safari, Android Chrome, and Desktop Web.

State Hygiene: You must manage online/offline states without corrupting local data or creating "fake" success states.

Performance: Sync operations must be asynchronous and never block the main UI thread. The app must remain responsive while syncing.

CORE OBJECTIVE
Users in the field frequently switch network states (e.g., toggle flight mode) or encounter "lie-fi" (connected but unreachable backend). The automatic background sync is insufficient. We need robust, manual controls to force real data synchronization and UI refreshes without requiring a full browser reload.

PART 1: THE SYNC LOGIC (ENGINE) REQUIREMENTS
Before building UI, ensure the core sync function handles these requirements precisely. Let's call this function: performFullSyncAction().

When performFullSyncAction() is triggered:

A) ROBUST NETWORK REACHABILITY CHECK (NON-NEGOTIABLE)
Do not rely solely on browser status. We must confirm backend connectivity.

Browser Check: First, check navigator.onLine.

If false, immediately proceed to Step A3 (Treat as Offline).

Backend Ping (The "Lie-Fi" Check):

If browser is true, perform a fetch request to a lightweight backend endpoint (e.g., /api/health or a simple timestamp endpoint).

Set a strict, short timeout (e.g., 3–5 seconds).

If Ping Succeeds (200 OK): Proceed to Step B (Online Sync Cycle).

If Ping Fails or Times Out: Proceed to Step A3 (Treat as Offline).

Treat as Offline Action:

Abort any further API attempts.

Trigger a re-render of current UI views from local persistent storage (IndexedDB cache).

Return status: OFFLINE_MODE.

B) THE ONLINE SYNC CYCLE (OPTIMIZED)
Set UI State: Set global app state to isSyncing: true.

PUSH (Upstream with Error Classification): Identify pending local mutations from the offline queue and attempt to send them to the backend. Crucially, classify any errors:

Network Errors / Timeouts (5xx, Failed to fetch): Keep the item in the queue, mark for retry later.

Permanent Failures (4xx, 401 Auth, 403 Forbidden, Validation Error): Mark the queued item as BLOCKED/FAILED. Do not retry endlessly. Surface a warning to the user (e.g., "3 items failed to sync and require attention").

PULL & REFRESH (Downstream): Once the push phase is complete (regardless of individual item successes/failures), immediately invalidate and refetch ALL main application queries to ensure the UI reflects the absolute truth from the server. This explicitly includes:

Herd lists and counts (Total, Rams, Ewes, Lambs).

Dashboard metrics.

Recently visited lists.

Finalize:

Set global app state to isSyncing: false.

Return status: SYNC_COMPLETE or SYNC_PARTIAL_ERROR based on the Push phase results.

PART 2: UI IMPLEMENTATION
A) Global Header: The "Refresh & Sync" Button
Placement: Top-right of the main application header, visible on all primary screens.

Design: A clean icon (e.g., circular arrows) + label "Refresh". Use standard app theme styling.

Behavior:

On Click: Call performFullSyncAction().

While Syncing: The button must be disabled and show a spinning loading state to prevent duplicate clicks.

Feedback:

If Result = SYNC_COMPLETE: Show subtle green toast: "Data synced."

If Result = OFFLINE_MODE: Show toast: "Backend unreachable. Reloaded local data."

If Result = SYNC_PARTIAL_ERROR: Show warning toast: "Sync complete, but some items failed."

B) Settings Screen: Data Controls Section
In the Settings area, add a new section titled "Data & Sync" with two distinct buttons placed side-by-side or stacked.

Button 1: "Sync Now (Online Only)"

Behavior: Initiates the full performFullSyncAction() workflow defined above, including the robust ping check.

Button 2: "Reload Data View"

Behavior: This is a UI un-sticker. It does not attempt any network calls. It forces the current view to re-read whatever data is currently available in the local cache/state manager.

Use Case: The user thinks the UI is stuck, even though data exists locally.

C) The Status Indicator Badge (Highly Recommended)
Add a small, non-intrusive badge in the header or footer.

🟢 Green Dot: Backend reachable + queues empty.

🟡 Yellow Dot: Backend reachable + items waiting in upload queue.

🔴 Red Dot: Backend unreachable (Offline or Ping Failed).

PART 3: GUARDRAILS (NON-NEGOTIABLE)
Debouncing: All manual sync buttons must be disabled while a sync is already in progress. Prevent parallel sync calls at all costs.

Storage Resilience (Incognito Mode): Wrap local storage persistence in try/catch blocks. If writing to IndexedDB fails due to quota/access errors (common in private mode), trigger a dismissible warning banner: "Browser storage restricted. Offline data may not be saved."

Truthful UI: Never show a "Saved" success message for an action that is only queued locally. Use accurate language: "Saved to device. Pending sync."
