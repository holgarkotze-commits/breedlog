We need to optimise storage, sync cost, and offline usability. Please implement the following changes carefully and confirm completion.

========================================
PART 1: IMAGE COMPRESSION (CRITICAL)
========================================

Problem:
Images are the heaviest data in the app. Uncompressed uploads increase sync time, storage cost, and hosting cost.

Goal:
Compress images BEFORE upload and BEFORE saving to IndexedDB, while keeping good visible quality suitable for livestock records.

REQUIREMENTS:
1) Implement CLIENT-SIDE image compression:
   - Compression must happen in the browser, before:
     - saving to IndexedDB (offline)
     - syncing to server (online)
   - Do NOT rely on server-side compression only.

2) Compression rules:
   - Target format: JPEG (or WebP if safely supported across browsers)
   - Max width: 1600–2000 px (maintain aspect ratio)
   - Quality: ~70–80% (visually clean, no obvious degradation)
   - Strip unnecessary metadata (EXIF where possible)

3) File size targets:
   - Typical photo should be reduced to ~200 KB – 500 KB
   - Must not upload multi-MB images unless absolutely necessary

4) UX requirements:
   - Compression must be automatic (no user action required)
   - Show a brief “Optimising image…” indicator during upload
   - Ensure upload still works offline (image stored locally first)

5) Storage:
   - Store ONLY the compressed image in IndexedDB
   - Sync ONLY the compressed version to the server
   - Original full-size image must NOT be stored

6) Confirm:
   - Faster sync
   - Reduced storage usage
   - No noticeable quality loss for animal identification

========================================
PART 2: OFFLINE IMAGE HANDLING
========================================

- When offline:
  - Allow image uploads
  - Store compressed images in IndexedDB
- When online:
  - Sync queued image uploads automatically
- Prevent data loss if app is closed before sync

========================================
PART 3: CLIENT-SIDE PDF GENERATION (OFFLINE)
========================================

We need confirmation and implementation (if not already done) that PDFs can be generated OFFLINE.

REQUIREMENTS:
1) PDF generation must be CLIENT-SIDE (browser-based):
   - No server dependency for PDF creation
   - Use a client-side PDF library (e.g. jsPDF or equivalent)

2) PDFs that must work OFFLINE:
   - Individual Animal Export PDF
   - Herd Export PDFs (Rams / Ewes / Lambs / Culled)
   - Records-based exports

3) Offline behavior:
   - User can generate PDF while fully offline
   - PDF is saved/downloaded locally on the device
   - If online later, PDF metadata (not file) may be logged/synced if needed

4) Layout consistency:
   - PDF layout must match current design
   - No degradation in formatting when offline

========================================
PART 4: CONFIRMATION REQUIRED
========================================

After implementation, please confirm explicitly:

1) Image compression is fully client-side
2) Average image upload size before vs after compression
3) Offline image upload + later sync works
4) PDFs generate successfully with ZERO internet connection
5) No regression in image quality for practical farm use

========================================
DO NOT:
- Store original large images
- Force users to manually compress
- Require internet for PDF exports
- Break existing offline sync logic

This optimisation is essential to keep the app fast, cheap to run, and reliable in low-connectivity farm environments.
