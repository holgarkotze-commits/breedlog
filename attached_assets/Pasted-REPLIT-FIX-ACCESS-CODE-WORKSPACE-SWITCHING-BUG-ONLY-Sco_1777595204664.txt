Confirmed bug:
In /api/beta/validate, the “Already activated” branch returns the existing activation user/workspace without checking whether the submitted invite code matches the existing activation’s inviteCodeId.

Result:
A phone that activated Code A can log out, enter Code B, and still receive Code A’s workspace/userId. Code B is effectively ignored.

This is wrong.

Correct behavior:

1. If same device enters the SAME code again:
   - allow re-login
   - return the same existing workspace
   - do not consume another slot

2. If same device enters a DIFFERENT code:
   - do NOT return the old activation automatically
   - validate the new submitted code
   - check available slot for that device type
   - switch the device/session to the new code’s workspace/account
   - return the new resolved workspace/userId
   - trigger client user-isolation logic because returned userId/workspace changed

3. If the new code has no slot available:
   - block clearly
   - do not keep showing the old workspace silently

4. If the new code is invalid/expired:
   - block clearly
   - do not keep showing the old workspace silently

5. IndexedDB/local cache:
   - do not globally wipe local data on logout
   - but when login returns a different resolved userId/workspace, existing ensureUserIsolation must clear/switch local IndexedDB so old herd data is not shown under the new code
   - confirm this path works

6. Seed script:
   - keep the fixed access-code resolver
   - seed dry-run must resolve --access-code to a real workspace/user, not the code string
